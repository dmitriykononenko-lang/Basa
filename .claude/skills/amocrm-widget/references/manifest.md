# manifest.json reference

`manifest.json` is the only descriptor the amoCRM widget loader reads at install time. Every other file is loaded by `script.js` at runtime. So this file controls: *where* the widget renders, *what* settings UI it shows, *which* API scopes it has, and *how* the marketplace presents it.

Save it as UTF-8, no BOM, valid JSON (trailing commas will silently break installs).

## Top-level skeleton

```json
{
  "widget": { ... },
  "locations": [ ... ],
  "settings": { ... },
  "dp": { ... },
  "advanced": { ... },
  "salesbot_designer": { ... }
}
```

`widget` and `locations` are required. The others are required only for the corresponding feature.

## The `widget` block

```json
"widget": {
  "name": "widget.name",
  "short_description": "widget.short_description",
  "description": "widget.description",
  "version": "1.0.0",
  "interface_version": 2,
  "init_once": false,
  "locale": ["ru", "en", "es", "pt"],
  "installation": "y",
  "support": {
    "link": "https://example.com/support",
    "email": "support@example.com"
  },
  "code": "my_widget_xyz",
  "secret_key": "PASTE_FROM_DEV_CABINET"
}
```

Field-by-field:

- **`name`, `short_description`, `description`** — i18n keys, resolved at runtime against `i18n/<locale>.json`. Don't put literal strings here. `short_description` (max ~80 chars) appears in the marketplace tile; `description` is the full body shown on the widget page.
- **`version`** — semver string. Must be bumped on every re-upload, otherwise the cabinet will reject the archive. Use `1.0.0` for the initial submission.
- **`interface_version`** — must be `2` for current widgets. `1` is legacy and won't render some UI surfaces (`advanced_settings`, modern modal). Use 2 unless maintaining a legacy widget.
- **`init_once`** — if `true`, `init` and `bind_actions` are called only once per page load, even when navigating between cards via SPA routing. Default is `false` (called every time the location is entered). Set to `true` only when you cache state across cards intentionally; otherwise stale state leaks between deals.
- **`locale`** — array of locale codes for which `i18n/<code>.json` exists. The user's account language is matched first; falls back to first locale in the array. Always include at least `ru` and `en`.
- **`installation`** — `"y"` shows the activation form during install, `"n"` skips it (use this only for widgets with no settings — rare; even API-key widgets need the form).
- **`support.link`** / **`support.email`** — shown on the marketplace page. Both are required for public widgets.
- **`code`** — unique identifier registered in the developer cabinet. Lowercase, alphanumeric + underscores. Must match what the cabinet assigned to your integration, or install will fail with "widget not found".
- **`secret_key`** — generated by the cabinet when you create the integration. Paste it verbatim. For private widgets, this stays in the archive; for marketplace widgets, the cabinet re-signs the archive on submission.

## The `locations` array

`locations` is the master list of UI surfaces the widget paints. Each entry is a string. Some entries embed a `-0` or `-1` suffix that controls whether the right-side card panel is shown (1) or only the JS context is loaded (0).

| Location key | Where it paints | Suffix meaning |
| --- | --- | --- |
| `lcard-1` / `lcard-0` | Lead card | 1 = panel shown, 0 = JS-only |
| `ccard-1` / `ccard-0` | Contact card | same |
| `comcard-1` / `comcard-0` | Company card | same |
| `cuscard-1` / `cuscard-0` | Customer card (loyalty) | same |
| `llist-1` / `llist-0` | Lead list view | 1 = action button row, 0 = silent |
| `clist-1` / `clist-0` | Contact list | same |
| `comlist-1` / `comlist-0` | Company list | same |
| `card_sdk` | Card SDK mount point (modern Vue-based cards) | no suffix |
| `settings` | Settings page tile + activation form | no suffix |
| `advanced_settings` | "Advanced settings" link on the widget tile after activation | no suffix |
| `digital_pipeline` | Pipeline automation action picker | no suffix |
| `lead_sources` | "Sources of leads" section — for chat / messenger widgets | no suffix |
| `catalogs` | Custom catalogs / lists pages | no suffix |
| `ai_agent` | AI agent tools (newer surface) | no suffix |
| `mail_card` | Inside the email composer | no suffix |
| `chats` | Chats sidebar (Chats API integrations) | no suffix |
| `sms` | SMS provider integration | no suffix |
| `everywhere` | Widget JS is bootstrapped on every page (use sparingly — slows the app) | no suffix |

Example: a widget that adds a panel to lead and contact cards, has a settings UI, and registers a DP action:

```json
"locations": [
  "lcard-1",
  "ccard-1",
  "settings",
  "digital_pipeline"
]
```

Common mistake: writing `"lcard": 1` (object) instead of `"lcard-1"` (string with suffix). The latter is correct.

## The `settings` object

Declares the fields shown in the activation form (or in the settings modal if you re-open settings later). Each entry is keyed by a field id; the value is a config object.

```json
"settings": {
  "api_key": {
    "name": "settings.api_key",
    "type": "text",
    "required": true
  },
  "api_secret": {
    "name": "settings.api_secret",
    "type": "pass",
    "required": true
  },
  "responsible_user": {
    "name": "settings.responsible_user",
    "type": "users",
    "required": false
  },
  "mode": {
    "name": "settings.mode",
    "type": "list",
    "values": {
      "sync": "settings.mode_sync",
      "async": "settings.mode_async"
    },
    "default": "sync"
  },
  "webhook_url": {
    "name": "settings.webhook_url",
    "type": "custom",
    "template": "settings_webhook"
  }
}
```

Supported `type` values:

- **`text`** — single-line plain input.
- **`pass`** — password input (masked).
- **`list`** — dropdown; declare `values` (id → i18n key) and optional `default`.
- **`users`** — picker over all account users; returns user id.
- **`users_lp`** — picker that supports "responsible user" semantics (includes the "as on lead" option).
- **`custom`** — render-your-own; references a key in the i18n that maps to a Twig template snippet you ship in `templates/`. Use this when none of the built-in types fit (multi-row table, slider, color picker, etc.). The custom template can include any HTML; you wire its behavior in `bind_actions`.

Field values are read in `script.js` via `self.get_settings()` (returns an object keyed by field id).

`required: true` blocks the install button until the field is filled. For `custom` fields, the loader can't enforce required, so validate in `bind_actions` and reject the save manually.

## The `dp` block (Digital Pipeline)

Required if `locations` includes `digital_pipeline`. Declares how the widget shows up as a DP action and what config fields its `dpSettings` modal exposes by default.

```json
"dp": {
  "settings": {
    "template_id": {
      "name": "dp.template_id",
      "type": "list",
      "values": { "1": "dp.tpl_welcome", "2": "dp.tpl_followup" }
    },
    "send_to": {
      "name": "dp.send_to",
      "type": "users_lp"
    }
  }
}
```

These are *default* fields; you can render arbitrary additional fields in the `dpSettings` callback's modal. Whatever the user picks is POSTed to your backend at the DP webhook endpoint (configured in the dev cabinet) when the pipeline step fires. The payload includes `widget_code`, `account_id`, `subdomain`, the lead/contact id, and the field values.

Required asset: `images/logo_dp.png` at **174×109 px**. Wrong dimensions = the DP action appears with a broken image.

## The `advanced` block

Optional. If you set `locations` to include `advanced_settings`, you can declare fields here that appear on the "advanced settings" page (accessed via a link on the widget tile after install). Use it for settings that don't need to be set at install time but the admin should be able to tweak later — webhook URLs, sync intervals, debug flags.

```json
"advanced": {
  "sync_interval": {
    "name": "advanced.sync_interval",
    "type": "list",
    "values": { "60": "advanced.every_minute", "300": "advanced.every_5_min" },
    "default": "300"
  }
}
```

Read advanced values via `self.get_settings()` same as regular settings; they're merged into one object.

## The `salesbot_designer` block

Only if the widget exposes a Salesbot action (custom step in the bot designer).

```json
"salesbot_designer": {
  "my_action": {
    "handler_code": "my_widget_action",
    "params": {
      "message_text": { "type": "text", "name": "sb.message_text" },
      "delay_seconds": { "type": "text", "name": "sb.delay_seconds" }
    }
  }
}
```

`handler_code` is the unique action id; `params` declares the fields the bot author fills in when dragging the action into a flow. When the flow runs, amoCRM POSTs to your `/salesbot/handler` endpoint with the params. The backend response controls the next step.

## OAuth flag

For widgets that talk to the amoCRM REST API on behalf of the user (not just to a third-party service), add at the top level:

```json
"oauth": "Y"
```

This signals the cabinet that the integration has a `client_uuid` and `client_secret` and uses the OAuth 2.0 flow. The widget gets access to the account's tokens via `AMOCRM.constant('account')` inside the JS context. Without this flag, OAuth refresh won't work even if your cabinet integration is configured.

## Marketplace-specific fields

Public widgets (submitted to the marketplace) also need:

- **`free`** — `"Y"` or `"N"`. Whether the widget is free.
- **`countries`** — array of country codes where the widget is available (`["RU", "BY", "KZ"]`, or `["*"]` for global).
- **`category`** — marketplace category code. The developer cabinet shows the authoritative dropdown when you create the integration; the values it currently accepts (May 2026) cluster around: `"telephony"`, `"messengers"`, `"sms"`, `"email_marketing"`, `"chatbots"`, `"forms"`, `"website"`, `"analytics"`, `"automation"`, `"accounting"`, `"finance"`, `"documents"`, `"sales"`, `"marketing"`, `"call_tracking"`, `"crm_integrations"`, `"ai"`, `"other"`. Always confirm the exact spelling against the cabinet dropdown — a typo silently breaks marketplace filtering even though the manifest validates.

These are reviewed by the marketplace team; submission rejections almost always cite missing or misleading values here. Make `description` match what the widget actually does — vague or inflated copy is the #1 rejection reason.

## Validation checklist before zipping

- Manifest is valid JSON (`python -m json.tool manifest.json` exits zero).
- `code` matches the developer-cabinet integration code exactly.
- `secret_key` is pasted from the cabinet, not invented.
- `version` is higher than the last uploaded version (semver; the cabinet rejects `1.0.0` on the second upload).
- `interface_version: 2`.
- Every i18n key used (`name`, field labels, list values) exists in every locale file declared in `locale`.
- Every `locations` entry maps to a callback/template you actually implemented.
- For `digital_pipeline`: `images/logo_dp.png` is exactly 174×109 px.
- For `oauth: "Y"`: integration in cabinet has matching redirect URI.
