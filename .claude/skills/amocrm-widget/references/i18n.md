# i18n — localization

amoCRM widgets must externalize all user-facing strings to `i18n/<locale>.json`. Hardcoded strings cause marketplace rejections (the review checks for them) and break for international users.

## File layout

```
i18n/
├── ru.json
├── en.json
├── es.json    # Kommo LATAM market
└── pt.json    # Kommo BR market
```

The locales you ship must match the `locale` array in `manifest.json`. If `manifest.locale = ["ru", "en"]`, you need exactly `i18n/ru.json` and `i18n/en.json`. Extra files are ignored; missing ones break that locale.

Every key that appears in one locale must appear in all of them. Mismatched keys = blank UI for users of the missing locale.

## JSON structure

Free-form nested objects. Use a flat first level by concern (settings, card, dp, errors, etc.):

```json
{
  "widget": {
    "name": "Мой виджет",
    "short_description": "Короткое описание",
    "description": "Полное описание для маркетплейса."
  },
  "settings": {
    "api_key": "API-ключ",
    "api_secret": "Секрет",
    "save": "Сохранить",
    "help_text": "Получите ключ в личном кабинете."
  },
  "card": {
    "title": "Информация",
    "no_data": "Нет данных",
    "refresh": "Обновить"
  },
  "dp": {
    "template_id": "Шаблон",
    "tpl_welcome": "Приветствие",
    "tpl_followup": "Догон"
  },
  "errors": {
    "required": "Обязательное поле",
    "network": "Ошибка сети"
  }
}
```

Keep keys lowercase + underscores. Avoid Cyrillic / Hangul / emoji in keys themselves — they break some path resolvers.

## How references resolve

### From `manifest.json`

When a field in manifest points to `"settings.api_key"`, the loader looks up `i18n[currentLocale].settings.api_key`. If missing, falls back to the first locale in `manifest.locale`. If still missing, shows the literal key string.

### From `script.js`

```js
self.i18n('settings').api_key        // returns "API-ключ" in ru, "API key" in en
self.i18n('settings.api_key')        // same, dotted notation
self.i18n('card')                    // returns the whole `card` object — useful to pass into templates
```

`self.i18n(section)` returns the section object. If you need a single string deeper than two levels:

```js
self.i18n('errors').network          // ok
self.i18n('errors.network')          // also ok
```

Don't chain into a missing path — you'll get `undefined`. Guard:

```js
(self.i18n('errors') && self.i18n('errors').network) || 'Network error'
```

### From `templates/*.twig`

The `langs` variable is the full language dict for the current locale:

```twig
<p>{{ langs.card.title }}</p>
<button>{{ langs.settings.save }}</button>
```

This is always available in every template render — you don't need to pass it explicitly.

## Locale resolution order

amoCRM picks the locale to use based on (in order):

1. The widget user's account language (set in amoCRM account settings).
2. The first locale in `manifest.locale` that has a matching JSON file.
3. English fallback.

You cannot override this from the widget — amoCRM controls it. If you need to ship per-user-language behavior (e.g., a Russian admin viewing an English account), use `self.system().language` to detect and branch in JS.

## `self.set_lang(obj)`

Sometimes you need to inject translations at runtime — e.g., the user picks "currency: EUR" in settings and you want to show "€" instead of "$". Use `set_lang`:

```js
self.set_lang({ card: { currency_symbol: '€' } });
```

This merges into the current language dict so subsequent `self.i18n('card').currency_symbol` and `langs.card.currency_symbol` in templates resolve to `€`. Doesn't persist — re-runs on each widget instantiation.

## Common pitfalls

- **Literal `settings.api_key` shown in UI.** The JSON file is missing, malformed, or the key doesn't exist. Validate with `python -m json.tool i18n/ru.json`.
- **Different keys in different locales.** The user switches account language and sees half the UI broken. Use a diff tool or a CI check to ensure key parity.
- **Translation includes HTML.** Putting `<a href>...</a>` inside i18n strings works in Twig (templates render raw by default), but in JS contexts you'll see escaped HTML. If you need rich text, ship plain text in i18n and wrap markup in the template.
- **Currency / date formatting hardcoded.** Don't translate dates as strings. Format in JS with `Intl.DateTimeFormat(self.system().language)`.
- **Long strings in `widget.short_description`.** Marketplace UI truncates after ~80 chars. Keep it tight.

## When to add a new locale

Add a new file to `i18n/`, copy keys from `en.json`, translate, and add the locale code to `manifest.locale`. Bump `version` and re-upload. No JS changes needed unless you want UI strings outside the standard flow.

## Translation workflow tip

Keep `en.json` as the master. Use a side-by-side diff in code review to verify each new key landed in every locale. For larger widgets, a CI script that fails if `ru.json` is missing a key from `en.json` is worth its weight in support tickets.
