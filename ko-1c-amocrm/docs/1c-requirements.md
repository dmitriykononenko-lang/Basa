# Требования к 1С-стороне интеграции 1С ↔ amoCRM

Исполнитель middleware: F5  
Версия 1С: 8.3.18.1208, самописная конфигурация, веб-клиент

---

## 1. Направление 1С → amoCRM (push-webhook)

Middleware принимает данные по HTTP. 1С-программист реализует отправку
HTTP-запроса из 1С при создании и изменении документа «Заказ».

### Endpoint

```
POST https://<middleware-host>/webhook/1c
Content-Type: application/json
X-Auth-Token: <WEBHOOK_SECRET>   ← значение предоставит F5
```

### Тело запроса (JSON)

```json
{
  "order_number":      "M00008597",
  "created_at":        "2025-06-10T09:00:00",
  "export_date":       "2025-06-15",
  "export_period":     "9-15",
  "status":            "Новый",
  "cancel_reason":     "",
  "last_name":         "Иванов",
  "first_name":        "Иван",
  "middle_name":       "Иванович",
  "phone":             "+79001234567",
  "cargo_type":        "Товар",
  "volume":            3.5,
  "weight":            120.0,
  "delivery_address":  "г. Москва, ул. Ленина, 1",
  "notes":             "Хрупкий груз",
  "amocrm_lead_id":    null
}
```

**Поля:**

| Поле | Тип | Обязательное | Описание |
|------|-----|:---:|---------|
| `order_number` | string | ✓ | Номер заказа, уникален |
| `created_at` | ISO 8601 datetime | — | Дата создания заказа |
| `export_date` | ISO 8601 date | — | Дата вывоза (`YYYY-MM-DD`) |
| `export_period` | string | — | Период вывоза, напр. `"9-15"` |
| `status` | string | ✓ | Текущий статус заказа |
| `cancel_reason` | string | — | Заполнять при статусе «Отменён» |
| `last_name` / `first_name` / `middle_name` | string | — | ФИО клиента по отдельности |
| `phone` | string | — | Телефон в любом формате |
| `cargo_type` | string | — | Тип груза (поле «Товар») |
| `volume` | float | — | Объём, м³ |
| `weight` | float | — | Вес, кг |
| `delivery_address` | string | — | Адрес доставки / поле «Данные заявки» |
| `notes` | string | — | Примечания |
| `amocrm_lead_id` | int или null | — | Передавать, если 1С уже хранит ID сделки |

### Ответ middleware

```json
{"ok": true, "lead_id": 123456, "order_number": "M00008597", "message": "synced"}
```

**Рекомендация:** сохранять `lead_id` из ответа в реквизит документа «Заказ»
(например, реквизит `АмоCRM_ID`, тип Число). При следующих изменениях заказа
передавать его в поле `amocrm_lead_id` — это ускоряет поиск и исключает
лишние запросы к amoCRM.

### Когда отправлять запрос

- При записи нового документа «Заказ»
- При изменении любого из полей, перечисленных выше
- При изменении статуса

**Идемпотентность гарантирована на стороне middleware:** повторная отправка
одного и того же `order_number` не создаёт дубль сделки.

## 2. Направление amoCRM → 1С (обратная синхронизация)

Middleware вызывает HTTP-сервис 1С при изменении этапа сделки в amoCRM.

### Что нужно реализовать в 1С

HTTP-сервис (встроенный REST 1С) с маршрутом:

```
POST /hs/amocrm/v1/status
```

База публикуется стандартным образом (Apache / IIS). URL целиком:

```
http://<1с-сервер>/<имя-базы>/hs/amocrm/v1/status
```

### Аутентификация

Basic Auth: логин/пароль пользователя 1С с правом на запись документов.
Передаётся в заголовке `Authorization: Basic <base64(login:password)>`.

### Тело запроса от middleware (JSON)

```json
{
  "amocrm_lead_id": 123456,
  "order_number":   "M00008597",
  "new_status":     "Подтверждён"
}
```

### Ожидаемый ответ от 1С

Успех:
```json
{"ok": true}
```

Ошибка (HTTP 4xx / 5xx):
```json
{"error": "Заказ M00008597 не найден"}
```

### Логика обработки на стороне 1С

1. Найти документ «Заказ» по `order_number` (или по реквизиту `АмоCRM_ID` если передан `amocrm_lead_id`)
2. Сопоставить `new_status` со статусом заказа 1С (таблица соответствия — от KO:AGENCY)
3. Записать новый статус
4. Вернуть `{"ok": true}`

---

## 3. Сетевые требования

- HTTP-сервис 1С должен быть доступен с IP-адреса VPS middleware
- Если 1С находится за NAT/VPN — согласовать проброс порта или VPN-туннель с F5
- Middleware работает по HTTPS; 1С-сервис может быть HTTP (в пределах закрытой сети)

## 5. Тестирование

F5 предоставит:
- URL и токен тестового middleware-стенда
- Тестовую коллекцию запросов (curl / Postman)

Приёмочные сценарии:
1. Создать заказ → в amoCRM появилась сделка + контакт + задача
2. Изменить статус в 1С → этап сделки обновился в amoCRM в течение 2 минут
3. Изменить этап сделки в amoCRM → статус заказа обновился в 1С в течение 2 минут
4. Повторно отправить заказ → дубль сделки не создан
