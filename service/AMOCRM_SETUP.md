# Подключение к боевой AmoCRM

Пошаговая инструкция для администратора. На выходе — рабочая интеграция с
вашим AmoCRM-аккаунтом: токены сохранены, автосоздание проектов работает по
вебхукам, метрики по задачам собираются.

## 0. Что должно быть готово до начала

- Поднятый сервис Basa, доступный из интернета по **HTTPS**.
  Это обязательно: AmoCRM не даёт `redirect_uri` на `http://` и не шлёт вебхуки
  на нешифрованный URL. На время разработки используйте Cloudflare Tunnel,
  ngrok или Cloudflare за вашим VPS — главное, чтобы был стабильный публичный URL.
  Допустим, у вас домен `basa.example.com`.
- Учётная запись администратора в AmoCRM с правом создавать интеграции.
- Доступ к `.env` сервиса и возможность перезапустить контейнер.

## 1. Создание интеграции в AmoCRM

1. Залогиньтесь в свой AmoCRM-аккаунт.
2. **Настройки → Интеграции → + Создать интеграцию**.
3. Выберите тип **«Доступно мне» (приватная)** — это long-lived
   integration по ТЗ §4.1.
4. Заполните карточку:
   - **Название:** Basa
   - **Описание:** Учёт работы аналитиков и выплат
   - **Логотип:** опционально
   - **Ссылка на интеграцию:** `https://basa.example.com/`
   - **Redirect URI:**
     ```
     https://basa.example.com/api/v1/amo/oauth/callback
     ```
     Важно: этот URI должен **точно** совпадать с тем, что вы потом запишете в
     переменную `AMO_REDIRECT_URI` в `.env`. Лишний слэш в конце ломает OAuth.
5. Откройте вкладку **«Права доступа»** и поставьте галки:
   - Чтение и изменение сделок
   - Чтение и изменение контактов (Amo требует для `with=contacts`)
   - Чтение и изменение задач
   - Чтение пользователей
6. Сохраните. Скопируйте появившиеся:
   - **ID интеграции** → это `AMO_CLIENT_ID`
   - **Секретный ключ** → это `AMO_CLIENT_SECRET`
   - **Код авторизации** — нам **не нужен**, мы получим свежий через redirect-flow.

## 2. Настройка вебхуков в AmoCRM

В той же интеграции откройте вкладку **«Webhook»** и добавьте подписку:

- **URL:** `https://basa.example.com/api/v1/amo/webhooks`
- **События:**
  - Сделки: добавление, изменение, удаление
  - Задачи: добавление, изменение, выполнение
- Сохраните.

Опционально (рекомендуется после первой проверки) — добавьте IP-whitelist в
Basa:

```bash
# из админки SPA → Настройки → amo_webhook_allowed_ips,
# либо через API:
curl -X PUT https://basa.example.com/api/v1/settings/amo_webhook_allowed_ips \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{"ips":["185.39.196.0/24","185.39.197.0/24"]}'
```

Актуальные диапазоны AmoCRM смотрите в их документации — здесь приведён
типовой пример, который может устареть.

## 3. Заполнение `.env`

Откройте `service/.env` и заполните блок AmoCRM (генерация ключей —
см. `.env.example`):

```bash
AMO_CLIENT_ID=<скопированный ID интеграции>
AMO_CLIENT_SECRET=<скопированный секрет>
AMO_REDIRECT_URI=https://basa.example.com/api/v1/amo/oauth/callback
AMO_BASE_URL=https://<ваш-поддомен>.amocrm.ru
```

`AMO_BASE_URL` — это **именно ваш поддомен**, на котором у вас открыт AmoCRM
(например, `https://mycompany.amocrm.ru`), а не `www.amocrm.ru`. От него
зависят и URL согласия, и URL обмена токенами.

Перезапустите контейнер:

```bash
docker compose up -d --build app worker
```

## 4. Согласие пользователя (OAuth)

1. Откройте веб-интерфейс Basa (`https://basa.example.com/`), залогиньтесь под
   администратором.
2. Вкладка **AmoCRM** → должна показаться карточка с состоянием:
   - **Env настроен:** Да
   - **Подключено:** Нет
3. Нажмите **«Подключить AmoCRM»**.
4. Откроется страница AmoCRM с просьбой подтвердить доступ. Нажмите
   **«Разрешить»**.
5. AmoCRM перенаправит обратно на `https://basa.example.com/api/v1/amo/oauth/callback`.
   Если всё прошло успешно, в ответе вы увидите:
   ```json
   {"status": "ok", "expires_at": "2026-..."}
   ```
6. Вернитесь во вкладку **AmoCRM** в Basa, перезагрузите страницу — теперь
   **Подключено: Да**.

Если получили **«Invalid or expired OAuth state»** — это означает, что
callback пришёл без `state`-параметра или с не тем, который мы сохранили.
Чаще всего причина — лишняя итерация (открыли start, ушли куда-то ещё,
вернулись по старой ссылке). Просто начните заново с шага 3.

## 5. Тест связи

В админке Basa → AmoCRM → **«Проверить связь»**. Сервис вызовет
`GET /api/v4/users` от вашего токена. Ожидаемый ответ:

```
Связь есть. Видим пользователей: N
```

Если получили `502 Bad Gateway: AmoCRM API error 401 …`, значит токены
протухли или scope недостаточен — переподключитесь.

## 6. Маппинг пользователей AmoCRM → аналитики Basa

Узнайте свой `amo_user_id` в Amo: **Настройки → Пользователи**, в URL
профиля. Затем в Basa создайте/отредактируйте аналитика:

```bash
PATCH /api/v1/analysts/<uuid>
{
  "amo_user_id": 1234567
}
```

Или через SPA → **Аналитики**.

## 7. Маппинг этапов воронки → действия

Узнайте `pipeline_id` и `status_id` ваших этапов:

```bash
curl https://basa.example.com/api/v1/amo/oauth/ping -X POST \
  -H "Authorization: Bearer $ADMIN_TOKEN"
# и затем — напрямую:
# GET /api/v4/leads/pipelines — этого эндпоинта в нашем API нет,
# но AmoClient умеет; временно можно через python REPL внутри контейнера.
```

Стандартное соответствие из ТЗ §4.4 (id заменить на свои):

| Этап в Amo                | Действие в Basa         |
|---------------------------|-------------------------|
| Договор подписан          | `start_project`         |
| Работа сдана              | `mark_done`             |
| Оплачено клиентом         | `mark_ready_for_payout` |
| Закрыто и не реализовано  | `cancel`                |

Сохраните через настройки:

```bash
PUT /api/v1/settings/amo_status_map
{
  "12345": "start_project",
  "12346": "mark_done",
  "12347": "mark_ready_for_payout",
  "12348": "cancel"
}
```

## 8. Первый пуш

Откройте сделку в AmoCRM, переведите её на этап «Договор подписан». Через
несколько секунд:

1. В Basa → **Журнал AmoCRM** появится новое событие `leads[update]` с
   `processed: true`.
2. В **Проекты** появится новый проект с этой сделкой.

Если проект не появился, проверьте:
- В журнале нет ли `error` у записи. Если есть — нажмите «Посмотреть»,
  изучите payload и стек.
- Связь `amo_user_id` ответственного с аналитиком установлена (см. шаг 6).
- Маппинг `amo_status_map` содержит **именно тот** `status_id`, в который
  переехала сделка.
- Если запись со статусом «Ошибка», можно нажать «Переобработать» после
  исправления маппинга.

## 9. Pull-синхронизация (страховка)

Раз в час имеет смысл подтягивать изменения за последние 24 часа на случай
потерянных вебхуков (ТЗ §2.3). Самый простой способ — внешний cron на хосте:

```cron
# каждый час — сделки и задачи за последние 24ч
*/60 *  * * *  curl -fsS -X POST https://basa.example.com/api/v1/amo/sync/run   -H "Authorization: Bearer $BASA_TOKEN"
5    *  * * *  curl -fsS -X POST https://basa.example.com/api/v1/amo/sync/tasks -H "Authorization: Bearer $BASA_TOKEN"
# раз в сутки — широкая сверка за 30 дней
30   3  * * *  curl -fsS -X POST "https://basa.example.com/api/v1/amo/sync/run?since=$(date -u -d '30 days ago' +%FT%TZ)" -H "Authorization: Bearer $BASA_TOKEN"
```

Внутрипроцессный планировщик (rq-scheduler / APScheduler) добавим отдельным
шагом, если внешний cron окажется неудобен.

## 10. Алерты на ошибки

В админке SPA → **Журнал AmoCRM** видна карточка «Ошибок за последний час».
По ТЗ §9.2 порог — 10 за час. Если хотите смотреть программно:

```bash
GET /api/v1/alerts/status?threshold=10
```

## Чек-лист готовности

- [ ] Сервис на HTTPS с публичным доменом.
- [ ] Интеграция в AmoCRM создана, redirect URI совпадает с `.env`.
- [ ] Webhook в AmoCRM настроен на `/api/v1/amo/webhooks`.
- [ ] `.env` заполнен, контейнер перезапущен.
- [ ] OAuth-подключение прошло, «Подключено: Да».
- [ ] «Проверить связь» возвращает список пользователей.
- [ ] Аналитики в Basa имеют `amo_user_id`.
- [ ] `amo_status_map` в settings — реальные id ваших этапов.
- [ ] Cron для pull-синков заведён.
- [ ] (Опционально) IP-whitelist для вебхуков включён.
