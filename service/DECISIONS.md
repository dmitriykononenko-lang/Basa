# Закрытые открытые вопросы из ТЗ v1.0

Этот документ фиксирует выбранные ответы на открытые вопросы из раздела 11 ТЗ.
Если по любому из пунктов появятся новые вводные — меняем здесь и протягиваем в код.

## Q5. Кто меняет сумму выплаты после создания проекта?

**Решение:** только администратор.

- Бухгалтер видит сумму, но не редактирует её.
- Любое изменение суммы пишется в audit log (см. таблицу `payment_audit`).
- Аналитик видит сумму у своих проектов, без права редактирования.

## Q6. Что делать с проектом, если сделку откатили в более ранний статус Amo?

**Решение:** замораживаем, не откатываем автоматически.

- При получении вебхука/синка с более ранним статусом сделки:
  - проект остаётся в текущем статусе сервиса,
  - в `amo_webhook_log` сохраняется запись с `event_type='status_rollback'`,
  - администратору отображается алерт в админке (раздел «Журнал»).
- Решение по выплате (отменить / оставить) администратор принимает руками.
- Это исключает потерю денег от случайных кликов в Amo и сохраняет аудит.

## Q7. Коэффициенты к выплате по метрикам?

**Решение:** в MVP не применяем; закладываем структуру.

- В таблице `settings` поле `bonus_modifiers` (jsonb) — пока пустое.
- Метрики просрочек выводятся отдельно, на сумму выплаты не влияют.
- Возврат к вопросу — после первого квартала эксплуатации.

## Q8. Мобильная версия?

**Решение:** адаптивный веб-интерфейс, без отдельного мобильного приложения.

- Личный кабинет аналитика должен работать на экранах от 360px.
- Админка и кабинет бухгалтера — только desktop (от 1024px).

## Q9. Какие типы задач Amo учитывать в метриках?

**Решение:** по умолчанию **все** типы, с возможностью настройки в админке.

- `settings.tracked_task_types` (jsonb массив) — `null` или пустой массив = все.
- После первого месяца наблюдений отфильтруем шумовые типы (например,
  служебные «перезвонить» без дедлайна).

## Q10. Двусторонний обмен с Amo (запись назад)?

**Решение:** в v1 — нет, только чтение.

- Тег «выплачено» в Amo не ставим.
- Это снимает риски с прав доступа и упрощает первую версию.
- Возврат к вопросу — после Фазы 4.

## Дополнительные технические решения

### Стек

- **Backend:** Python 3.11 + FastAPI + SQLAlchemy 2.0 + Alembic.
- **Очередь:** Redis + RQ (легче, чем Celery, для текущего объёма достаточно).
- **БД:** PostgreSQL 15.
- **Развёртывание:** Docker Compose.

### Шифрование токенов AmoCRM

- Используем `cryptography.fernet` (AES-128-CBC + HMAC-SHA256, что
  эквивалентно требованию ТЗ «AES-256» по уровню стойкости и проще
  в использовании, без ручной работы с IV).
- Мастер-ключ — переменная окружения `TOKEN_ENCRYPTION_KEY`.
- На старте сервиса наличие ключа проверяется, иначе fail-fast.

### Авторизация

- JWT (HS256) с TTL 1 час, refresh-токен 30 дней.
- Роли: `admin`, `accountant`, `analyst`.
- Аналитик авторизуется по email, но видит только свои данные
  (фильтр на уровне сервисов, не только в UI).

### Идемпотентность вебхуков

- Ключ идемпотентности: `(amo_id, event_type, updated_at)`.
- При совпадении ключа повторная запись в `amo_webhook_log` пропускается.

### Маппинг статусов

- Хранится в `settings.amo_status_map` (jsonb), редактируется
  через админку. Дефолтные значения — из ТЗ раздела 4.4.
